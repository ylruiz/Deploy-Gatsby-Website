Everyone knows that web pages show in the browser windows text, images, and videos, but how is that possible? The process behind the information that a website shows to its visitors is generated by simple, text-only documents known as source documents.

In order to do that, you only need to use HTML - .html files - and add style to the HTML elements by using CSS - .css files. You can also add some behaviors to the elements of the page using a scripting language programming called JavaScript - .js files.

A quick introduction to HTML, CSS, and JavaScript

HTML stands for HyperText Markup Language. It is a markup language that tells the browser how to display the content by using tags. The tags describe each element on the page, such as headings, paragraphs, emphasized text, and hyperlinks. Other tags add information about the document, for instance, its title, media like images and videos, and widgets for form inputs.

The version of HTML that we use nowadays is HTML5. You can read about how we got here on Wikipedia. What matters to us - or can affect us - of all that information is that there are two organizations in charge of HTML somehow: W3C (World Wide Web Consortium) and WHATWG (Web Hypertext Application Technology Group). While W3C takes care of HTML5 standards, WHATWG focuses on maintaining and evolving HTML.

CSS stands for Cascading Style Sheets. It allows us to add visual styles to our pages. In other words, it helps us to indicate to the browsers how the documents .html should be presented to the users.

JS stands for JavaScript. It is a scripting language that allows us to add a behavior to the component on a page. It is an essential ingredient in modern websites.

To sum up, whereas HTML provides the structure and CSS style sheet allows us to change the way things look for default in a browser, JS lets us have control about how things work.

Some concepts you need to know

Every day the web matures, and the number of devices increases. As a consequence, our job as web designers and developers get more complicated due to we have too many unknown elements in our medium. For example, users will see our websites but we will never know what browsers or devices will be used, what the devices' resolutions will be, how the connection to internet will be, etc.

Consequently, it is our job to know some important concepts and take care of them when we are making a website:

  • The multitude of devices

    Nowadays, multiple devices can use the web: computers, laptops, phones, watches, TVs, refrigerators, gaming systems, and so on. You should keep that in mind when you are developing a website because you can be doing that on a computer or a laptop, but the users can see it on bigger or smaller devices.

  • Web standards

    A good way to deal with the existed diversity of devices is to follow the standards documented by W3C. This can ensure you that you are being as consistent as possible with all the standards-complain browsers, which make the web standards your friend.

    In addition, you can verify if you can use some HTML tag, HTML attribute, css property, etc by using the web resource Can I Use.

  • Progressive enhancement

    As you should know, many browsers can be used depending on the users' preferences. This means that we need to face the fact that there is a wide range of browser capabilities. One strategy for dealing with unknown browser capabilities is progressive enhancement.

    Progressive enhancement is a strategy in web design that focuses on web content first, separating the presentational semantics from the content. This enables that the page styles may be activated depending on the browser or the quality of the Internet connection. So, progressive enhancement allows everyone to access the elemental content and functionalities of a web page, while the users with additional browser capabilities or better Internet connection can access the enhanced version of the web page.

  • Responsive Web Design

    Responsive Web Design (RWD) is a strategy for dealing with unknown screen size. It consists of providing the appropriate layouts to devices based on the size of the browser window, which is also called viewport.

    To make a web page responsive, you have an HTML document for all devices. However, you apply different style sheets depending on the screen size, which means that you do all the magic by using CSS.

    RWD is a relevant part of modern websites because it makes the experience of their visitors more delightful.

  • Accessibility

    People use different browsers and many devices to access the web. They also can access the web in many different ways: keyboard, mouse, fingertips, voice commands, screen readers, Braille display, screen magnifier, joysticks, foot pedals, modified mice or keyboard, and so on. So, you must design for accessibility, which helps users with disabilities or users with less-than-optimum browsing experiences. For instance, people with vision, mobility, auditory and cognitive impairment can benefit from it. You can obtain more information about this subject on the site started by W3C to address the need for making the web accessible for everyone: Web Accessibility Initiative (WAI).

  • Site performance

    Site performance is critical, regardless of how users access a website. Users expect a website to load in under 2 seconds; in another case, they will probably leave the site and never come back. It is also likely that if your website is a slowpoke, Google will not show it on the first screens of results.

    All the above mentioned makes clear why it is so important to take down your site performance to milliseconds. Of course, there are many things you can do to improve site performance: optimizing images, streamline HTML markup, minimizing HTML and CSS documents, keeping JS to a minimum, adding scripts in a way that they can load in parallel with other pages without blocking rendering, don't load unnecessary assets, and reducing the number of HTTP requests.

    There are also tools for testing site performance: WebPage Test, Google's PageSpeed Insights, and GTMmetrix.

Starting a Technical Documentation Project

The best way to learn something is by practicing. That is the reason why I would like to start an easy project where I just what to focus on using HTML, CSS, and JS. It will be a Web Design Technical Documentation Project with an orientation to how we use those technologies. Here is an initial prototype of this project.

It is time now to follow the first steps to start this project.

Step 1: Create a folder called web-design.

Step 2: Inside that folder, create a file called index.html

Step 3: Open the index.html file and type the following code, which is just a basic document structure. Remember that you should NOT copy any code because you will not learn when you do that.

     1   <!DOCTYPE html>
     2   <html>
     3       <head>
     4           <meta charset="utf-8">
     5           <title>Web Design Documentation</title>
     6       </head>
     7       <body>
     8           Page content goes here.
     9       </body>
    10   </html>

The line number 1 is just a document type declaration, which is also known as DOCTYPE declaration. This DOCTYPE identifies the document as written in HTML5.

The entire document is contained within an html element, which is also called root element due to it contains all the elements in the document, and it may NOT be contained by any other element. Inside the html element, the document is divided into a head element and a body element.

The head element is a container for metadata (data about the HTML document), which are not displayed. Metadata typically defines the document title, character set, styles, scripts, and other meta information. In this case, the head element contains a meta element (line 4) that specifies the character encoding used in the document as Unicado version UTF-8. Then, it has a title element (line 5), which defines the title of the document. The title element is mandatory according to the HTML specifications.

The body element contains everything that we what to show up in the browser window.

You should always validate your HTML document by using, for instance, validator.w3.org.

Step 4: Create a remote repository, for example by using GitHub - if you do not have an account, then you should create one first.

Step 5: Open a command prompt window, and move to the folder where you have created your project. When you get there, enter the following commands:

    > git init
    > git remote add origin your_remote_repository
    > git add .
    > git commit -m "Initial commit"
    > git push -u origin master

If you do not have GIT installed on your computer, you need to install it first. I recommend you to follow the steps described on its official website.
